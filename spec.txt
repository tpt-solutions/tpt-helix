1. Purpose
TPT Helix is a next-generation web platform built from first principles. It is not a browser — it is the successor to the browser.
The modern web has become an accidental operating system: bloated, insecure, and economically captured by surveillance capitalism. Thirty years of backward compatibility have turned the web platform into an archaeological dig of accumulated complexity. JavaScript, designed in 10 days for form validation, now underpins the global economy. The browser has become a universal runtime that nobody designed and nobody can replace.
TPT Helix replaces this. It is:
A WASM-native application platform that renders web content without a JavaScript engine
An AI-accelerated migration system that ports existing web applications to native WASM
A capability-secure runtime where applications declare what they need, not what they can take
A memory-efficient engine that uses 60–80% less RAM than Chromium for equivalent workloads
TPT Helix does not compete with Chrome. It obsoletes the architecture that made Chrome necessary.
2. Goals
2.1 Primary Goals
| #  | Goal                               | Success Criteria                                                                                 |
| -- | ---------------------------------- | ------------------------------------------------------------------------------------------------ |
| G1 | **WASM-Native Runtime**            | Execute web applications compiled to WASM without a JavaScript interpreter in the hot path       |
| G2 | **AI-Accelerated Migration**       | Port 80% of common web application patterns from JS to Rust/WASM autonomously within 12 months   |
| G3 | **Capability Security**            | Replace origin-based security with capability-based access control; no ambient authority         |
| G4 | **Memory Efficiency**              | Achieve ≤150 MB base RAM for a single complex application (e.g., video player, document editor)  |
| G5 | **Content-Addressed Distribution** | Replace location-based URLs with content-addressed identifiers (IPFS-style) for immutable assets |
2.2 Secondary Goals
| #   | Goal                     | Success Criteria                                                                                                 |
| --- | ------------------------ | ---------------------------------------------------------------------------------------------------------------- |
| G6  | **Legacy Compatibility** | Run existing JavaScript applications via embedded lightweight interpreter (QuickJS/Boa) during transition period |
| G7  | **Cross-Platform**       | Native builds for Linux, macOS, Windows, Android, iOS, and embedded (Raspberry Pi, ESP32-class where feasible)   |
| G8  | **Developer Experience** | Provide source-language choice: TypeScript-subset, Rust, or Zig → compile to WASM with WIT interfaces            |
| G9  | **Offline-First**        | All applications function without network; sync is explicit and user-controlled                                  |
| G10 | **Open Standard**        | WIT interface definitions and capability protocols are open specifications, not proprietary                      |
2.3 Non-Goals
| #   | Non-Goal                                                    | Rationale                                                                                                                            |
| --- | ----------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------ |
| NG1 | **Pixel-perfect Chrome compatibility**                      | The web platform spec is 50M+ lines of accumulated complexity. Helix targets a clean subset + migration path, not bug-for-bug parity |
| NG2 | **Support for `eval`, `document.write`, `with` statements** | These break static analysis and capability tracking. Legacy code using them runs in the JS compatibility layer                       |
| NG3 | **Surveillance-based monetization**                         | No telemetry, no fingerprinting, no ad-tech integration. Revenue from licensing, support, and enterprise deployment                  |
| NG4 | **Immediate consumer market dominance**                     | Target enterprise, embedded, and controlled-content ecosystems first. Network effects follow utility                                 |

5. Capability Model
5.1 Principles
No ambient authority. A module has no access to anything not explicitly granted.
User grants, not site requests. The user decides what an application receives, not the application.
Fine-grained, revocable. Capabilities are per-resource and can be revoked at any time.
Composable. Capabilities can be delegated between modules.
5.2 Capability Types (WIT Interfaces)
// Example WIT interface definitions

interface network {
    record request {
        method: string,
        url: string,
        headers: list<tuple<string, string>>,
        body: option<list<u8>>,
    }

    record response {
        status: u16,
        headers: list<tuple<string, string>>,
        body: list<u8>,
    }

    fetch: func(req: request) -> result<response, string>
}

interface storage {
    get: func(key: string) -> option<list<u8>>
    set: func(key: string, value: list<u8>) -> result<_, string>
    delete: func(key: string) -> result<_, string>
}

interface dom {
    record element-id {
        id: u64,
    }

    create-element: func(tag: string) -> element-id
    set-text: func(el: element-id, text: string)
    set-attribute: func(el: element-id, name: string, value: string)
    append-child: func(parent: element-id, child: element-id)
    on-click: func(el: element-id, handler-id: u64)
}

// Authored WIT lives in `crates/helix-wit/wit/helix.wit`. The host
// (`helix-runtime`) *exports* these interfaces (a guest *imports* them); the
// package is `helix:runtime@0.1.0`, generating Rust modules under
// `helix::runtime::{network, storage, dom}`. Errors are surfaced as plain
// `string`s. Callbacks (`on-click`) cross the boundary as opaque `handler-id`s
// rather than first-class function values.

    record response {
        status: u16,
        headers: list<tuple<string, string>>,
        body: list<u8>,
    }

    // Performs an HTTP-flavored request. Fails with `error` when the module
    // lacks the network capability for `req.url` or the fetch cannot be made.
    fetch: func(req: request) -> result<response, error>
}

interface storage {
    // Shared error type surfaced by capability-checked operations.
    type error = string

    record key-value {
        key: string,
        value: list<u8>,
    }

    get: func(key: string) -> option<list<u8>>
    set: func(key: string, value: list<u8>) -> result<_, error>
    delete: func(key: string) -> result<_, error>
}

interface dom {
    record element-id {
        id: u64,
    }

    create-element: func(tag: string) -> element-id
    set-text: func(el: element-id, text: string)
    set-attribute: func(el: element-id, name: string, value: string)
    append-child: func(parent: element-id, child: element-id)
    // Registers the click handler identified by `handler-id`. `handler-id` is
    // allocated and owned by the guest; the runtime invokes it (via the
    // capability broker / wasmtime) when the event fires. Modeled as a
    // guest-managed id rather than a WIT function reference so the host stub
    // can be exercised without a live guest component.
    on-click: func(el: element-id, handler-id: u64)
}

// Two worlds wrap the interfaces above (see `wit/helix.wit`):
//   * world `helix`        — the guest world; guests *import* the interfaces.
//   * world `helix-host`   — the host world; the Helix Runtime *exports* them
//                            (generating the `Guest` traits the host fills in).

interface media {
    record video-config {
        codec: string,  // "h264", "vp9", "av1"
        width: u32,
        height: u32,
        bitrate: u32,
    }

    create-player: func(config: video-config) -> result<player-handle, error>
    play: func(handle: player-handle)
    pause: func(handle: player-handle)
    seek: func(handle: player-handle, time-ms: u64)
}
5.3 Grant Flow
User opens application
    │
    ▼
Application declares required capabilities:
    "I need network:fetch to api.example.com"
    "I need storage:local (10MB)"
    "I need dom:full"
    "I need media:video (720p max)"
    │
    ▼
Helix Runtime presents grant request to user
    │
    ▼
User approves/denies/modifies each capability
    │
    ▼
Runtime creates restricted handles, passes to WASM module
    │
    ▼
Module operates within granted capabilities only
Any attempt to exceed → trap/abort

8. Security Model
8.1 Threats Addressed
| Threat              | Traditional Web            | Helix                                           |
| ------------------- | -------------------------- | ----------------------------------------------- |
| XSS                 | CSP, sanitization          | No JS injection vector; WASM is typed           |
| CSRF                | SameSite cookies, tokens   | No ambient authority; capabilities are explicit |
| Prototype pollution | Object.freeze, linting     | No prototype chain in WASM                      |
| Memory corruption   | ASLR, sandbox              | Rust + WASM memory safety                       |
| Spectre/Meltdown    | Site isolation (processes) | Capability isolation (cheaper, finer-grained)   |
| Supply chain        | npm audit, SRI             | Content-addressed dependencies                  |

11. Glossary
Table
Term	Definition
WASM	WebAssembly — binary instruction format, sandboxed, portable
WIT	WebAssembly Interface Types — typed interfaces between WASM modules and hosts
WASI	WebAssembly System Interface — POSIX-like APIs for WASM
Capability	Unforgeable token granting access to a specific resource
Ambient Authority	Implicit access to resources based on identity (e.g., origin) rather than explicit grant
Content-Addressed	Identifying data by its cryptographic hash, not its network location
DASH	Dynamic Adaptive Streaming over HTTP — video streaming protocol
CDM	Content Decryption Module — DRM component for protected media
EME	Encrypted Media Extensions — browser API for DRM
AOT	Ahead-of-Time compilation — compile before execution
JIT	Just-in-Time compilation — compile during execution
